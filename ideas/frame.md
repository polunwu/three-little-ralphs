## Pre-work：v2 改進方向景觀

Draft 中浮現出四個有訊號的方向，另有兩個被明確延後或留作問題：

| 選項 | 內容 | 誰受益 | 訊號強度 |
|------|------|--------|----------|
| **A. 降低啟動摩擦** | 自動複製 settings、明確啟動順序（Judge → Reviewer → Executor） | 每次重新啟動工作流的使用者 | 直接寫出，是具體操作步驟 |
| **B. 提升角色可見性** | 每個角色在工作前看 git diff，掌握當前變更上下文 | Reviewer / Judge 需要知道 Executor 改了什麼 | 直接寫出，針對角色協作的核心痛點 |
| **C. 結構分層（/agent, /skill）** | 角色定義放 `/agent`，可呼叫的功能放 `/skill` | 工作流維護者、未來擴充角色或技能時 | 直接寫出，反映 v1 結構混用的問題 |
| **D. 完成狀態可追蹤** | Executor 完成後標記 `[x]`，另存獨立完成紀錄檔 | 想知道工作進度的使用者 | 兩條相關筆記，一起指向「使用者看不清楚完成了什麼」 |

延後項（作者明確說「先不用」）：
- **打包安裝到不同專案**：功能有價值，但目前工作流還在驗證階段，過早打包會鎖死未穩定的介面。

待決定項（沒有足夠訊號納入本輪）：
- **git worktree / branch**：以問號出現，代表作者尚未確認是否需要、如何做。不納入本輪 frame。

**為什麼這四個方向合在一起 frame：** 這四個方向都在解決同一個根本問題——v1 工作流「可以跑起來」，但**對使用者不透明**，也**對角色不友善**。啟動步驟不明確、角色缺乏上下文、結構讓角色與技能混用、完成狀態不可見，是同一層問題的四個面向，適合在一輪 shaping 中一起處理。

## Problem

- 啟動工作流需要手動記憶步驟（複製 settings、知道角色啟動順序），沒有明確流程或保護（直接引用 draft 啟動筆記）
- Reviewer 和 Judge 在工作時不知道 Executor 改了什麼，只能靠任務描述推測，缺乏直接的變更上下文（直接引用 draft「要叫三個角色看 git diff」）
- `/workflow` 目錄中角色定義與可呼叫功能混放，沒有結構分層，擴充時容易混亂（直接引用 draft `/agent, /skill` 分層筆記）
- 使用者沒有辦法快速看到「到目前為止完成了哪些任務」，只能看 state.json 或 git log（直接引用 draft 完成紀錄筆記）

## Outcome

- 工作流有明確的啟動程序，步驟可被執行不需要記憶
- 每個角色在進入工作前，能取得足夠的變更上下文（git diff 或等效資訊）
- 角色與技能在目錄結構上明確分層，新增角色或技能時有清楚的放置位置
- 完成的任務在任務列表中有標記，並有獨立的完成紀錄檔供使用者查閱

## Less about

- 讓工作流跑得更快或更自動
- 改變三角色的核心狀態機邏輯（v1 已驗證，本輪不動）

## More about

- 讓工作流對人更透明：使用者知道發生什麼、角色知道當前狀態
- 讓結構對未來擴充更友善：分層讓每個新增的角色或技能有明確的歸屬
